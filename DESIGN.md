This technical document outlines the architecture for **SwiftTopology**, a high-performance system observability tool. By bypassing high-level abstractions like Aya and instead utilizing **libbpf**, **CO-RE (Compile Once – Run Everywhere)**, and manual **Rust FFI**, we achieve maximum control over the kernel-user boundary.

---

# Technical Design Document: SwiftTopology
**Project Code:** ST-01  
**Lead Organization:** SwiftLogic Systems  
**Architecture Style:** Deep-System eBPF with Rust/C FFI

## 1. Vision & Objectives
SwiftTopology aims to provide a real-time, interactive Terminal User Interface (TUI) that maps hardware topology (Caches, NUMA, Cores) and overlays live performance metrics (IPC, Cache Misses, Thread Migration) using eBPF.

The "Deep" approach ensures:
*   **Kernel Transparency:** Using `vmlinux.h` directly to access internal kernel structures.
*   **Minimal Footprint:** Leveraging `libbpf` (the C standard) for the smallest possible binary size.
*   **Safety & Aesthetics:** Using Rust for the logic layer and Ratatui for a modern, fluid UI.

---

## 2. System Architecture

The system is divided into three distinct layers:

### A. The Kernel Layer (eBPF / C)
Written in restricted C, this layer lives in the kernel and performs the data collection.

*   **Header Strategy:** Use `vmlinux.h` (generated via `bpftool vmlinux dump scan > vmlinux.h`) to eliminate dependencies on system headers and enable CO-RE.
*   **Probes:**
    *   `tracepoint/sched/sched_switch`: To track which PID is on which CPU core.
    *   `perf_event`: To read Hardware Performance Counters (L3 Misses, Instructions).
*   **Data Transport:** BPF Maps (specifically `BPF_MAP_TYPE_RINGBUF` for events and `BPF_MAP_TYPE_PERCPU_ARRAY` for core-local stats).

### B. The Bridge Layer (libbpf-rs / FFI)
This layer acts as the glue, loading the eBPF bytecode and mapping C structs to Rust.

*   **Skeleton Generation:** Use `libbpf-cargo` to generate a Rust "skeleton" of the C-eBPF program.
*   **Shared Memory:** Define a shared C struct in a header file included by both the eBPF C code and the Rust code (via `bindgen`).

### C. The Presentation Layer (Rust / Ratatui)
The user-space logic and the TUI.

*   **State Management:** An async loop that polls the BPF Ring Buffer.
*   **Topology Discovery:** Directly reading `/sys/devices/system/cpu/` or using `libhwloc` via FFI to build the initial hardware tree.
*   **Rendering:** A Ratatui loop drawing the "Circuit Board" layout based on the data in the local state.

## 3. Implementation Blueprint

### Step 1: The eBPF Program (`swifttopology.bpf.c`)
```c
#include "vmlinux.h"
#include <bpf/bpf_helpers.h>
#include <bpf/bpf_tracing.h>

struct cpu_stats {
    u64 total_cycles;
    u64 instructions;
    u64 cache_misses;
    u32 current_pid;
};

struct {
    __uint(type, BPF_MAP_TYPE_PERCPU_ARRAY);
    __uint(max_entries, 256); // Support up to 256 cores
    __type(key, u32);
    __type(value, struct cpu_stats);
} cpu_data_map SEC(".maps");

SEC("tracepoint/sched/sched_switch")
int handle_sched_switch(struct trace_event_raw_sched_switch *ctx) {
    u32 cpu_id = bpf_get_smp_processor_id();
    struct cpu_stats *stats = bpf_map_lookup_elem(&cpu_data_map, &cpu_id);
    if (stats) {
        stats->current_pid = ctx->next_pid;
    }
    return 0;
}
```

## 4. Build Pipeline
To support this "Deep" approach, the build process must be strictly orchestrated:

1.  **Metadata Extraction:** `bpftool` extracts `vmlinux.h` from the running kernel.
2.  **Clang Compilation:** The `swifttopology.bpf.c` is compiled to LLVM bitcode and then to BPF ELF.
3.  **Skeleton Generation:** `libbpf-cargo` takes the ELF and generates Rust bindings.
4.  **Rust Compilation:** Cargo compiles the user-space code, linking against the static `libbpf`.
5.  **Final Result:** A single, statically-linked binary with no external dependencies (except `libc`).

## 5. Strategic Research Value (SwiftLogic Systems)
Open-sourcing **SwiftTopology** under this architecture serves as a high-authority proof of work. 

*   **Demonstrates FFI Mastery:** Shows the company can bridge modern languages (Rust) with legacy kernel standards (C/libbpf).
*   **Visualizes the Latency Wall:** By showing L3 misses pulsing in the TUI, SwiftLogic proves they understand why software "feels" slow despite low CPU usage.
*   **Market Positioning:** Positioned as the "Interactive successor to lstopo," attracting users from high-performance sectors (Finance, Telco, GameDev).
